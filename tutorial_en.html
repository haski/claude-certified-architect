Chapter 01 · Claude API

Fundamentals of Model Interaction

Everything an agent does sits on top of one request–response call. Master the shape of that call — and the fact that the model remembers nothing between calls — and the rest of the exam falls into place.

The Claude API is stateless. Each request is independent; the model has no server-side memory of prior turns. To hold a conversation, you resend the full history every time.

1.1The shape of a request

Every call to the Messages API is a JSON object. These fields recur throughout the exam:

messages.request.jsonjson
{
  "model": "claude-sonnet-4-6",
  "max_tokens": 1024,
  "system": "You are a helpful assistant.",
  "messages": [
    {"role": "user", "content": "Hi!"},
    {"role": "assistant", "content": "Hello!"},
    {"role": "user", "content": "How are you?"}
  ],
  "tools": [],
  "tool_choice": {"type": "auto"}
}
FieldRole
modelWhich model — e.g. claude-opus-4-6, claude-sonnet-4-6, claude-haiku-4-5
max_tokensMaximum tokens in the response (not the context limit)
systemThe system prompt — behaviour, role, constraints. Passed separately, not inside messages
messagesThe full conversation history — you must resend it every call
toolsDefinitions of callable tools
tool_choiceTool-selection strategy (Chapter 2)

1.2Statelessness, visualised

Because nothing persists, a "conversation" is really a growing transcript you carry forward. This single fact explains a cluster of exam answers about lost memory and rising cost.

Each request carries the entire history — so long conversations cost more and run slower (Ch 11).

1.3Message roles & stop_reason

The messages array uses three roles: user, assistant (the model's prior replies, included when you resend), and tool results (which appear as a tool_result content block, not an explicit role). Every response carries a stop_reason that drives your control flow:

stop_reasonMeaningWhat you do
end_turnThe model finishedShow the result — the loop ends
tool_useThe model wants a toolRun it, append the result, call again
max_tokensHit the token capResponse is truncated; raise the limit
stop_sequenceA stop string appearedHandle per your app logic
Exam anchor

For agent loops, only two values matter: tool_use (keep looping) and end_turn (stop). The reliable completion signal is always stop_reason == "end_turn" — never parsed text, never an iteration counter.

1.4The system prompt

The system prompt defines context and behavioural rules. It is loaded once, applies throughout, and takes priority over user messages. Use it for role, constraints, and output format.

Watch for unintended tool associations

System-prompt wording can accidentally steer tool selection. An instruction like "always verify the customer" can push the model to over-call get_customer even when it is unnecessary. (This is exactly the trap in practice Q56.)

1.5The context window

The context window is the total text (in tokens) the model processes at once: system prompt + full history + tool definitions + tool results. Three failure modes recur on the exam:

Lost in the middle — models attend best to the start and end of long inputs. Put key info at the edges (Ch 11).
  1. Lost-in-the-middle: details buried in the middle of a long input get missed. Fix: place key facts at the start or end.
  2. Tool-result accumulation: a tool returning 40+ fields when 5 matter wastes most of the window. Fix: trim outputs (Ch 11).
  3. Progressive summarisation: numbers, percentages, and dates degrade into vague words ("about", "roughly") when history is compressed. Fix: a persistent facts block (Ch 11).

Key takeaways

  • The API is stateless — resend the full messages history on every call.
  • The system prompt is separate from messages and outranks user text.
  • stop_reason drives the agent loop: tool_use → continue, end_turn → stop.
  • The context window holds prompt + history + tools + results; mind lost-in-the-middle and summarisation loss.

Chapter 02 · Tools

Tools & tool_use

Claude doesn't run code — it requests a structured call and you execute it. The description you write is the single most important factor in whether it picks the right tool.

tool_use lets Claude call external functions. The model emits a structured request; your code runs it and returns a tool_result. This is also the most reliable way to get structured JSON out of Claude.

2.1Defining a tool

A tool is a name, a description, and a JSON-schema for its inputs:

tool.get_customer.jsonjson
{
  "name": "get_customer",
  "description": "Finds a customer by email or ID. Returns the customer profile (name, email, order history, account status). Use BEFORE lookup_order to verify identity. Accepts an email (user@domain.com) or a numeric customer_id.",
  "input_schema": {
    "type": "object",
    "properties": {
      "email": {"type": "string", "description": "Customer email"},
      "customer_id": {"type": "integer", "description": "Numeric customer ID"}
    },
    "required": []
  }
}

2.2The description is the routing mechanism

An LLM selects tools by their descriptions. This is the most-tested idea in Domain 2. Minimal descriptions like "Retrieves customer information" cause misrouting when tools overlap.

✗ Weak

analyze_content — "analyzes content and extracts key information"

analyze_document — "analyzes documents and extracts key information"

Near-identical → the model confuses them and misroutes ~45% of requests.

✓ Strong

Rename + differentiate: extract_web_results — "processes and returns information retrieved from web search and URLs."

Now each tool's purpose is unambiguous.

A good description states: what it does and returns · input formats with examples · edge cases and limits · when to use it vs. a similar tool.

Built-in vs. MCP tools

Agents may prefer built-in tools (Read, Grep) over MCP tools with similar functions. To win them over, strengthen the MCP description — highlight unique data or context the built-ins can't provide.

2.3The tool_choice parameter

Three strategies — looser to stricter. Pick the loosest that still gives the guarantee you need.
ValueBehaviourWhen
{"type":"auto"}Model decides: tool or textDefault for most cases
{"type":"any"}Must call some toolGuarantee structured output
{"type":"tool","name":"x"}Must call tool xForce a specific first step / ordering

2.4JSON schemas for structured output

Using tool_use with a JSON schema is the most reliable way to get structured output. It guarantees valid JSON and the required structure — but not semantic correctness (values can still be wrong).

extraction_schema.jsonjson
{
  "type": "object",
  "properties": {
    "category": {"type": "string", "enum": ["bug", "feature", "docs", "unclear", "other"]},
    "category_detail": {"type": ["string", "null"], "description": "Details if category = other/unclear"},
    "severity": {"type": "string", "enum": ["critical", "high", "medium", "low"]},
    "confidence": {"type": "number", "minimum": 0, "maximum": 1},
    "optional_field": {"type": ["string", "null"], "description": "Null if not found in source"}
  },
  "required": ["category", "severity"]
}

Schema design rules the exam loves:

  • Required only when always present. Required fields push the model to fabricate values when data is missing.
  • Nullable for maybe-absent data: "type": ["string", "null"] lets the model return null instead of hallucinating.
  • Enums with "other" + a detail string prevent losing data outside your categories.
  • Enum "unclear" — an honest "unclear" beats a confidently wrong category.

2.5Syntax vs. semantic errors

TypeExampleMitigation
SyntaxInvalid JSON, wrong field typetool_use + JSON schema eliminates these
SemanticTotals don't add up, value in wrong field, hallucinationValidation checks, retry-with-feedback, self-correction (Ch 6)

Key takeaways

  • Tool descriptions are the primary selection mechanism — make them rich and distinct.
  • Eliminate overlap by renaming and differentiating, not by adding classifiers.
  • tool_choice: auto (default) → any (force a tool) → forced tool (force order).
  • Schemas kill syntax errors; nullable fields + "other"/"unclear" enums prevent fabrication; semantic errors still need validation.

Check yourself · Chapter 2

Three questions on tool design and structured output.
Q1 An agent keeps calling get_customer for order questions. Both tools have minimal, similar descriptions. What is the most effective first step?
Correct: B. Descriptions are the model's primary selection mechanism — the lowest-effort, highest-impact fix. Classifiers are over-engineering; few-shot adds tokens without addressing the root cause; merging loses capability.
Q2 You must guarantee Claude returns structured data via one of several extraction tools — never plain text. Which setting?

Chapter 03 · Agent SDK

Building Agentic Systems

The agent loop, the coordinator–subagent topology, and hooks. This chapter feeds Domain 1 — the single biggest slice of the exam at 27%.

3.1The agentic loop

An agent doesn't just answer — it acts in a loop, deciding each next step from context and prior results. This is model-driven, unlike a hard-coded decision tree.

Loop while tool_use · stop on end_turn. Tool results are appended to history so the model can choose the next action.
Anti-patterns to avoid

Parsing assistant text for "Task completed" · using an arbitrary max_iterations as the primary stop condition · treating "the model produced text" as completion. The only reliable signal is stop_reason == "end_turn".

3.2AgentDefinition

agent.pypython
agent = AgentDefinition(
    name="customer_support",
    description="Handles returns and order issues",
    system_prompt="You are a customer support agent...",
    allowed_tools=["get_customer", "lookup_order", "process_refund", "escalate_to_human"],
    # For a coordinator, include the Task tool:
    # allowed_tools=["Task", "get_customer", ...]
)

allowed_tools follows the principle of least privilege — give each agent only the tools its role needs. Too many tools (18 instead of 4–5) measurably reduces selection reliability.

3.3Hub-and-spoke: coordinator & subagents

All communication flows through the coordinator — the hub owns observability, error handling, and routing.

The coordinator decomposes the task, dynamically selects subagents, delegates, aggregates & validates results, handles errors/retries, and reports to the user.

Critical principle — isolated context

Subagents do not inherit the coordinator's history. Required context must be explicitly passed in the subagent's prompt. They share no memory across calls, and all communication routes through the coordinator for observability and control.

3.4Spawning subagents with Task

The coordinator's allowed_tools must include "Task". Context passing is mandatory:

✗ No context

Task: "Analyze the document"

The subagent has nothing to work with.

✓ Full context

Task: "Analyze the following document. Document: [full text] Prior results: [...] Output format: [schema]"

Parallel spawning: a coordinator can emit multiple Task calls in one response — the subagents run concurrently. Use this to partition research and avoid duplication.

3.5Hooks: deterministic interception

Hooks intercept the agent lifecycle. PostToolUse transforms a tool result before the model sees it; a pre-call hook can block policy-violating actions.

hooks.pypython
@hook("PostToolUse")
def normalize_dates(tool_result):
    # Unix timestamp -> ISO 8601, "Mar 5, 2025" -> "2025-03-05"
    return normalized_result

@hook("PreToolUse")
def enforce_refund_limit(tool_call):
    if tool_call.name == "process_refund" and tool_call.args.amount > 500:
        return redirect_to_escalation(tool_call)   # deterministic block
When failure has financial, legal, or safety consequences — use a hook, not a prompt.

Key takeaways

  • Loop on tool_use, stop on end_turn — never on text or an iteration cap.
  • Coordinator owns decomposition, routing, aggregation, and error handling; subagents have isolated context.
  • Pass all needed context explicitly in the Task prompt; spawn multiple Tasks for parallelism.
  • Hooks give deterministic guarantees; prompts are only probabilistic. Critical rules → hooks.
Quiz

Check yourself · Chapter 3

Correct: C. Financial rules need deterministic enforcement. Hooks guarantee 100% compliance; prompts are only probabilistic.
Q2 A subagent receives Task: "Analyze the document" and produces nothing useful. Why?

Chapter 04 · MCP

Model Context Protocol

MCP is the open standard that plugs external systems into Claude. Know its three primitives, how servers are configured, and what a good error looks like.

4.1Three primitives

Tools act · Resources inform · Prompts template. Knowing which is which is a frequent exam discriminator.

An MCP server is a process implementing the protocol. On connection, all its tools are discovered automatically, and tools from all connected servers are available at once. As always, descriptions determine how they're used.

4.2Configuring servers: project vs. user

.mcp.jsonjson
{
  "mcpServers": {
    "github": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-github"],
      "env": { "GITHUB_TOKEN": "${GITHUB_TOKEN}" }
    }
  }
}
ScopeFileUse for
Project.mcp.json (repo root, in VCS)Team integrations — shared with everyone who clones
User~/.claude.json (home, not in VCS)Personal / experimental servers
Secrets via env vars

${GITHUB_TOKEN} substitution means the .mcp.json is version-controlled but the token never is. Each developer supplies their own via environment variable — the idiomatic team setup (practice Q44).

For standard integrations (Jira, GitHub, Slack), prefer existing community MCP servers; build your own only for unique, team-specific workflows.

4.3The isError flag & structured errors

When an MCP tool fails, it returns isError: true. How it describes the failure decides whether the agent can recover.

✓ Structured error

{
  "isError": true,
  "content": {
    "errorCategory": "transient",
    "isRetryable": true,
    "message": "Orders API timeout.",
    "attempted_query": "order_id=12345",
    "partial_results": null
  }
}

✗ Generic error

{
  "isError": true,
  "content": "Operation failed"
}

The agent can't tell whether to retry, re-query, or escalate.

Exam anchor — error categories

Return errorCategory (transient / validation / business / permission), isRetryable, and a human message. Transient = retry; validation = fix input; business = explain & offer alternative; permission = escalate. (Full table in Ch 10.)

4.4Resources as a "map"

Resources are data an agent reads for context without taking action: content catalogs, database schemas, documentation, issue summaries. Their advantage — the agent gets an immediate map of what exists, instead of burning exploratory tool calls to discover it.

Key takeaways

  • MCP primitives: Tools (act), Resources (read for context), Prompts (templates).
  • Project .mcp.json (team, VCS) vs. user ~/.claude.json (personal); secrets via ${ENV}.
  • Prefer community servers for standard integrations.
  • Structured errors (errorCategory, isRetryable, message, partials) enable recovery; generic errors don't.
Quiz

Check yourself · Chapter 4

Three questions on MCP.
Q1 Six developers each have a personal GitHub token. You want one shared MCP config without committing secrets. Best approach?
Correct: B. Project .mcp.json with env-var substitution is a single VCS source of truth while each developer supplies their own credential — no secrets committed.
Q2 Which is an MCP Resource rather than a Tool?
Correct: B. Resources are read-only context (schemas, docs, catalogs). The others perform actions, so they're Tools.
Q3 A subagent gets {"isError": true, "content": "Operation failed"}. What's the design flaw?
Correct: B. Without an error category, retryable flag, and context, the agent can't choose a recovery path. Structured errors fix this.
Chapter 05 · Claude Code

Configuration & Workflows

Domain 3 is 20% of the exam and lives almost entirely here: the CLAUDE.md hierarchy, path-scoped rules, skills, planning mode, and the CLI for CI/CD.

5.1The CLAUDE.md hierarchy

Three levels. The classic exam bug: team guidance placed at user level, so new teammates never receive it.
The most-tested CLAUDE.md mistake

Three developers' "always add error handling" rule works, but a new hire's doesn't — because it lives in their personal ~/.claude/CLAUDE.md, not the project .claude/CLAUDE.md. Fix: move it to the project file so everyone gets it (practice Q37).

5.2@path imports & .claude/rules/

CLAUDE.md can import other files with @path (no space, relative or absolute, max nesting depth 5) to stay modular:

CLAUDE.mdmarkdown
Coding standards are in @./standards/coding-style.md
Test requirements are in @./standards/testing-requirements.md
Overview in @README.md and dependencies in @package.json

For larger codebases, .claude/rules/ splits guidance into topic files, each with optional YAML frontmatter that scopes it to file globs:

.claude/rules/testing.mdyaml
---
paths: ["**/*.test.tsx", "**/*.test.ts"]
---

Tests must use describe/it blocks.
Use data factories instead of hardcoding.
Do not mock the database — use a test database.
Path-scoped rules vs. directory CLAUDE.md

A path rule loads only when Claude edits a matching file — saving context. Use .claude/rules/ with globs when a convention spans many directories (e.g. tests scattered next to code). Use a directory-level CLAUDE.md when it's tied to one folder. Globs win for cross-cutting patterns (practice Q40).

5.3Slash commands & skills

Slash commands are reusable prompt templates invoked via /name. In current Claude Code, .claude/commands/ and .claude/skills/ both create /name commands; skills add a SKILL.md with frontmatter.

.claude/skills/analyze/SKILL.mdyaml
---
context: fork
allowed-tools: ["Read", "Grep", "Glob"]
argument-hint: "Path to the directory to analyze"
---

Analyze the code structure in the specified directory.
Output a report on dependencies and architectural patterns.
FrontmatterEffect
context: forkRuns in an isolated subagent — verbose output won't pollute the main session
allowed-toolsRestricts the toolset (e.g. can't delete files if not allowed)
argument-hintPrompts for an argument when invoked without one
LocationScope
.claude/commands/ · .claude/skills/Project — in VCS, available to the whole team on clone
~/.claude/commands/ · ~/.claude/skills/User — personal, not shared. Overrides a project skill of the same name

Skill vs. CLAUDE.md: a skill is on-demand for a specific task (review, analysis, generation); CLAUDE.md is always-loaded standards.

5.4Planning mode vs. direct execution

Plan when the approach is uncertain or the blast radius is large; execute directly when the change is small and clear. Often: plan to discover, then execute.

The Explore subagent isolates verbose discovery output and returns only a summary — preventing context-window exhaustion in multi-phase tasks.

5.5The CLI for CI/CD

The -p / --print flag runs Claude Code non-interactively — the only correct way in a pipeline:

ci.shbash
# Non-interactive: process prompt, print to stdout, exit
claude -p "Analyze this pull request for security issues"

# Structured output for automated PR comments
claude -p "Review this PR" --output-format json --json-schema '{"type":"object"}'
Session context isolation

The same session that generated code is worse at reviewing it — it keeps its own reasoning and won't challenge itself. Use a fresh, independent instance for review. When re-reviewing after new commits, include prior findings and ask Claude to report only new/unresolved issues.

Other session controls: --resume <name> continues a saved session (risk: stale tool results if files changed); fork_session branches from shared context to compare approaches; /compact compresses history (risk: losing exact numbers); /memory edits CLAUDE.md for cross-session notes.

Key takeaways

  • CLAUDE.md: user (personal) · project (team, VCS) · directory. Team rules go in the project file.
  • .claude/rules/ with paths globs loads conventions only for matching files — best for cross-cutting patterns.
  • Skills: context: fork isolates output · allowed-tools restricts · argument-hint prompts. Project vs. user location decides sharing.
  • Plan for big/uncertain work, execute directly for small/clear; use -p for CI and a fresh instance for review.

Chapter 06 · Prompting

Prompt Engineering, Advanced

Few-shot examples, explicit criteria, chaining, the interview pattern, and validation loops. The engine room of Domain 4.

6.1Few-shot prompting

Few-shot means including 2–4 input/output examples to demonstrate expected behaviour. It beats prose because a vague instruction ("be more precise") is interpreted many ways, while an example shows the exact format and decision logic — and the model generalises the pattern rather than parroting it.

few_shot_ambiguous.txttext
Request: "My order is broken"
Action: get_customer -> lookup_order -> check status.
Why: "broken" may mean damaged; you need order details first.

Request: "Get me a manager"
Action: Immediately call escalate_to_human.
Why: explicit request for a human — do not try to solve.

Few-shot shines for ambiguous tool selection, output formatting, separating acceptable vs. problematic code, extraction across document formats, and informal measurements ("two handfuls of rice" → ~100g). Pair it with normalization rules in the prompt (dates → ISO 8601, "half" → 0.5) to prevent semantic drift.

6.2Explicit criteria beat vague instructions

✗ Vague

"Check code comments for accuracy. Be conservative — report only high-confidence findings."

✓ Explicit

"Flag a comment ONLY if: (1) it contradicts actual behaviour, (2) it references a non-existent symbol, (3) a TODO refers to an already-fixed bug. Do NOT flag stylistic or minor wording issues."

Define severity with examples too — CRITICAL (runtime failure), HIGH (security hole), MEDIUM (logic bug, no immediate impact), LOW (code quality). Vague guidance like "be more conservative" reliably underperforms concrete categories.

False positives erode trust

High false-positive rates in noisy categories (style, naming) make developers dismiss everything — including accurate security findings. The exam's fix: temporarily disable the noisy categories, keep high-precision ones, improve prompts, then re-enable (practice Q23/Q29).

6.3Prompt chaining

Break a complex task into focused steps. Reviewing 14 files at once causes attention dilution — deep analysis on some, shallow on others, contradictory verdicts. Instead: analyse each file individually, then run a separate cross-file integration pass.

Per-file passes give consistent depth; a final integration pass catches cross-file issues. Chaining suits predictable tasks; dynamic decomposition suits open-ended ones (Ch 8).

6.4Iterative refinement (Domain 3.5)

Three techniques for converging on the right output through iteration:

  • Concrete input/output examples — the single most effective way to communicate a transformation when prose is interpreted inconsistently. Show 2–3 representative pairs (this is the fix in sample Q31).
  • The interview pattern — before implementing, Claude asks clarifying questions. Useful in unfamiliar domains (fintech, healthcare), for non-obvious implications (cache invalidation, failure modes), and when multiple approaches depend on context the user hasn't stated.
  • Test-driven iteration — write the test suite first (covering expected behaviour, edge cases, and performance requirements), then iterate by feeding the test failures back to guide progressive improvement. The tests pin the target so each round measurably converges.
One message or several? Interacting vs. independent issues

When fixes interact (changing one affects another), give Claude all of them in a single detailed message so it can reason about them together. When issues are independent, iterate sequentially — one fix per round — to keep each change focused and easy to verify.

6.5Validation, retry-with-feedback & self-correction

When extracted data fails validation, retry with the original document, the previous incorrect extraction, and the specific error:

retry_feedback.txttext
Field 'total' = 150, but sum(line_items) = 145. Re-check values.
Retry helpsRetry won't help
Format errors (wrong date format)Information absent from the source
Structural errors (field misplaced)Required context is in another, unprovided document
Arithmetic inconsistencies (re-checkable)—

Pydantic is the exam's reference validator: structural checks (types, enums), semantic checks (custom validators for business rules), validate-retry loops, and JSON-Schema generation as a single source of truth. Self-correction extracts both a stated_total and a calculated_total, setting conflict_detected when they differ.

Track the trigger with detected_pattern

For review/extraction findings, add a detected_pattern field recording which code construct or rule triggered the finding. When developers dismiss findings, you can analyse these patterns to spot systematic false positives and tune the prompt — turning dismissals into a feedback signal rather than noise.

Key takeaways

  • Few-shot (2–4 examples) is the most reliable way to fix inconsistent format/behaviour — especially for ambiguous cases.
  • Explicit categorical criteria beat vague conservativeness; manage false positives by category.
  • Chain focused passes to beat attention dilution; per-file + integration pass for big reviews.
  • Iterative refinement: concrete examples · interview pattern · test-driven iteration; batch interacting fixes, sequence independent ones.
  • Retry-with-feedback fixes format/structure/arithmetic — not missing information.

Chapter 07 · Batches

Message Batches API

Half the cost, up to a day of latency, no mid-request tool calling. One trade-off table answers most batch questions on the exam.

The Batch API submits requests for asynchronous processing: 50% cheaper, processed within up to 24 hours (no latency SLA), no multi-turn tool calling (one request = one response), and a custom_id per request to correlate results.

Why batch can't do iterative tool calling

A "fire-and-forget" async model has no way to pause mid-request, run a tool, and feed the result back for Claude to continue. Any workflow needing multiple tool round-trips in one logical interaction is fundamentally incompatible (practice Q18).

7.2Batch vs. synchronous

If a developer is blocked on the result → synchronous. If it runs overnight or in bulk → batch.
TaskAPIWhy
Pre-merge PR checkSyncDeveloper is waiting; 24h unacceptable
Overnight tech-debt reportBatchNeeded by morning; 50% savings
Weekly security auditBatchNot urgent; 50% savings
Interactive code reviewSyncImmediate response required
Processing 10,000 documentsBatchBulk; savings significant

7.3custom_id, failures & SLA planning

custom_id links each result to its source document, so on failure you re-submit only the failures — never re-processing successes. A typical recovery: 100 docs → 95 succeed, 5 hit the context limit → identify by custom_id → split the long docs → re-submit just those 5.

SLA math

Need a result in 30 hours with up to 24h processing? Your submission window is 30 − 24 = 6 hours — submit no later than 24h before the deadline. For steady throughput, split into ~4-hour submission windows.

Key takeaways

  • Batch = 50% cheaper, up to 24h, no multi-turn tool calling, custom_id correlation.
  • Blocking/interactive → synchronous; overnight/bulk → batch.
  • Iterative tool-calling workflows can't use batch at all.
  • Re-submit only failed custom_ids; plan submission windows against the 24h ceiling.

Chapter 08 · Decomposition

Task Decomposition Strategies

Fixed pipelines vs. dynamic adaptation — and why splitting a big review into passes outperforms a single megaprompt.

8.1Fixed pipelines vs. dynamic decomposition

Fixed pipeline (prompt chaining)

Every step known up front:
Doc → metadata → extract → validate → enrich → output

Use for predictable, repeatable tasks needing stability (reviews, migrations).

Dynamic adaptive decomposition

Subtasks emerge from intermediate results: map structure → find untested modules → prioritise by risk → adapt when an external dependency appears.

Use for open-ended investigation where scope is unknown.

Quick rule

Predictable & repeatable → prompt chaining. Open-ended, each step depends on the last → dynamic decomposition. "Add tests to a legacy codebase" is dynamic; "review every PR the same way" is fixed.

8.2Multi-pass code review

For a PR touching 10+ files, a single pass over everything causes attention dilution: deep analysis on some files, shallow on others, missed obvious bugs, and contradictory verdicts (a pattern flagged in one file, approved in identical code in another).

Restructure into focused passes — analyse each file for local issues, then a separate integration pass for cross-file data flows. A bigger context window does not fix this; attention quality, not capacity, is the bottleneck.

Key takeaways

  • Fixed pipeline for predictable work; dynamic decomposition for open-ended investigation.
  • Big reviews → per-file passes + one integration pass.
  • Attention dilution, not context size, causes inconsistent multi-file reviews.

9.1Reliable triggers vs. false signals

✓ Reliable triggers

Explicit request ("get me a manager") → escalate immediately, don't try to solve

Can't make progress after reasonable attempts

Financial op above threshold (enforce via hook)

✗ Unreliable signals

9.2Escalation patterns

Frustration alone is not a manager request. Acknowledge → resolve → escalate only if the customer insists again.

9.3Structured handoff & confidence calibration

On escalation, pass a self-contained summary — the human sees only this, not the transcript: customer id/name, issue summary, order id, root cause, actions taken, recommended action, escalation reason.

Stratified sampling

For extraction systems, output field-level confidence and route low-confidence/ambiguous cases to humans. Audit even high-confidence outputs with stratified random sampling — a 97% aggregate accuracy can hide 40% errors on one document type. Analyse by type and field, not just overall.

Key takeaways

  • Escalate on: explicit request, policy gap, stuck, over-threshold financial, ambiguous identity.
  • Sentiment, self-rated confidence, and bespoke classifiers are unreliable triggers.
  • Acknowledge → resolve → escalate only on reiteration; never on first frustration.
  • Handoff summaries must be complete; audit with stratified sampling by type and field.

Categorise the error, recover locally when you can, and propagate structured context — never a generic "failed".

10.1Error categories

CategoryExamplesRetryableAgent action
TransientTimeout, 503, networkYesRetry with backoff
ValidationBad format, missing fieldNoFix input & retry
BusinessPolicy violation, thresholdNoExplain; offer alternative
PermissionAccess deniedNoEscalate

10.2Anti-patterns

Anti-patternWhy it's badCorrect approach
Generic "search unavailable"Coordinator can't decide recoveryReturn type, query, partials, alternatives
Silent suppression (empty = success)Hides failure as "no matches"Distinguish "no results" from "search failed"
Abort whole workflow on one failureLoses all partial resultsContinue with partials; annotate gaps
Infinite retries inside a subagentLatency & wasted resources1–2 local retries, then propagate
Access failure ≠ empty result

A timeout (access failure, needs a retry decision) is semantically different from "0 results" (a valid, informative answer). Conflating them is a classic wrong answer — and distinguishing them is a classic right one (practice Q12).

10.3Structured subagent error & coverage annotations

subagent_error.jsonjson
{
  "status": "partial_failure",
  "failure_type": "timeout",
  "attempted_query": "AI impact on music industry 2024",
  "partial_results": [{"title": "AI Music Report", "relevance": 0.8}],
  "alternative_approaches": ["Try a narrower query", "Use an alternative source"],
  "coverage_impact": "Not covered: AI impact on music production"
}

This gives the coordinator everything it needs to decide: retry with a modified query, use partials, delegate elsewhere, or continue and annotate the gap. In the final synthesis, mark each section's coverage — FULL vs. PARTIAL (search timeout) — so readers know where confidence is lower (graceful degradation with transparency).

Key takeaways

  • Categorise: transient (retry) · validation (fix input) · business (explain) · permission (escalate).
  • Recover locally for transient errors (1–2 retries), then propagate non-recoverable ones with partials.
  • Never return generic failures or silently suppress; never abort the whole workflow on one failure.
  • Distinguish access failures from valid empty results; annotate coverage in synthesis.

Chapter 11 · Context

Context Management in Production

Protect critical facts from summarisation, trim noisy tool output, and position information where the model actually reads it.

11.1Extract facts into a persistent block

History degrades during summarisation. Keep critical transactional facts in a structured block included in every prompt, outside the summarised history:

case_facts.txttext
=== CASE FACTS (updated as new facts appear) ===
Customer ID: CUST-12345    Order ID: ORD-67890
Order Date: 2025-01-15     Order Amount: $89.99
Issue: Damaged item on delivery
Customer Request: Full refund    Status: Pending manager approval
===
Why not just tune summarisation?

Summarisation inherently loses precise numbers, dates, and percentages. Raising a threshold or "asking it to preserve numbers" is fragile. A structured facts block guarantees the data survives regardless of how much history is compressed (practice Q54).

11.2Trim tool results & position-aware input

If lookup_order returns 40+ fields but you need 5, a PostToolUse hook keeps only what matters — conserving context and reducing noise. And because of lost-in-the-middle, structure aggregated input deliberately:

positioned_input.txttext
[KEY FINDINGS — at the top]      Found 3 critical vulnerabilities...
[DETAILED RESULTS — middle]      === auth.ts === ...  === db.ts === ...
[ACTION ITEMS — at the end]      Priority: fix auth.ts before merge.

A key-findings summary at the start (primacy) plus explicit section headings throughout directly mitigates the missed-middle problem (practice Q13).

11.3Scratchpads, subagent delegation & state persistence

In long investigations, write key findings to a scratchpad file — when context degrades or a new session starts, the agent consults it instead of re-discovering. Delegate verbose discovery to a subagent (Explore) that reads 15 files and returns one line, so the main agent keeps a clean window. Give subagents minimal context, a constrained allowed_tools, and ask for structured results — not raw dumps. For crash recovery, each agent exports its state to a known location and the coordinator loads a manifest on resume.

The subagent absorbs 15 files of noise and returns a one-line summary — the main context stays clean.

Key takeaways

  • Persist critical facts in a structured block outside summarised history.
  • Trim verbose tool output (PostToolUse hook) to relevant fields.
  • Position key findings at the start; use section headings to beat lost-in-the-middle.
  • Use scratchpads, subagent delegation, and exported state for long/large investigations.

Chapter 12 · Provenance

Preserving Provenance

Keep the claim-to-source link alive through synthesis, and handle conflicting data by annotating — never by silently picking a winner.

12.1Don't lose attribution

✗ Attribution lost

"The AI music market is estimated at $3.2B." — no source, no year.

✓ Provenance preserved

{claim, source_url, source_name, publication_date, confidence} — the claim travels with its evidence.

Require subagents to output claim → source mappings (URL, document name, quotes) so attribution survives aggregation.

12.2Conflicting data & dates

When two credible sources disagree, preserve both with attribution and let the coordinator reconcile — don't arbitrarily pick one:

conflict.jsonjson
{
  "claim": "Share of AI-generated music on streaming",
  "values": [
    {"value": "12%", "source": "Spotify Report 2024", "date": "2024-03"},
    {"value": "8%",  "source": "Industry Survey",      "date": "2024-07"}
  ],
  "conflict_detected": true,
  "possible_explanation": "Different methodology and time period"
}

Always include dates — without them, temporal differences get misread as contradictions ("A says 10%, B says 15%" vs. "A (2023) 10%, B (2024) 15% — likely +5% growth"). And render by content type: financial data → tables, news → prose, technical findings → lists, time series → chronological.

Key takeaways

  • Preserve claim → source → date through synthesis; require structured citations from subagents.
  • On conflict, keep both values with attribution and let the coordinator reconcile.
  • Dates prevent misreading temporal change as contradiction.
  • Render by content type rather than forcing one format.

Grep vs. Glob

The frequent confusion: Glob finds files by name/pattern; Grep searches file contents. "Find all functions named X" → Grep. "Find all .test.tsx files" → Glob.

13.2Incremental investigation & the Edit fallback
